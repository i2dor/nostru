NIP-352
=======

Bitcoin Silent Payment Address
-------------------------------

`draft` `optional`

## Summary

A Nostr event that publishes a Bitcoin Silent Payment address
discoverable via social identity, without cryptographically binding the
payment address to the social keypair.

## Motivation

BIP-352 Silent Payments allow a sender to derive a unique Bitcoin output
from a static payment code (`sp1...`). Existing proposals derive the
`sp1` address directly from the Nostr public key, permanently linking
social identity to payment identity. This has two consequences:

1. **No rotation.** The address is mathematically fixed to the keypair.
   Changing it requires rotating the Nostr identity.
2. **Surveillance linkage.** Anyone who knows the npub can compute the
   `sp1` address without the receiver's involvement or consent.

This NIP separates *discoverability* (finding a payment address through
social identity) from *derivation* (which keypair the address is
computed from). The result is a rotatable payment identity that senders
discover through the social graph but that does not have to be derived
from it.

## Event kind

**kind: 30352** — addressable, one per pubkey per network.

`30352 = 30000 (addressable range, NIP-01) + 352 (BIP-352)`.

Events in the `30000–39999` range are parameterized replaceable: for
each combination of `kind`, `pubkey`, and `d` tag value, relays keep
only the latest event. This allows separate, independently rotatable
addresses for mainnet and test networks.

## Tags

| Tag | Value | Required | Description |
|-----|-------|----------|-------------|
| `d` | `"mainnet"` \| `"signet"` \| `"testnet"` | yes | Network identifier. Together with `kind` and `pubkey` this forms the replaceable key. |
| `sp` | BIP-352 address | yes | The static payment address (`sp1...`). |
| `payment_pubkey` | 33-byte compressed pubkey (hex) | no | The scan+spend public key from which `sp` is derived, if different from the event author. Allows client-side verification. |

Content: empty or a human-readable note (not machine-parsed).

## Example

```json
{
  "kind": 30352,
  "pubkey": "<social pubkey hex>",
  "content": "",
  "tags": [
    ["d", "mainnet"],
    ["sp", "sp1qqgmccn46xdv3dfts7e7y9k3m5ln7xa4rqe4rqdvnn9vp6kfzr6lqzgq..."],
    ["payment_pubkey", "02a1b2c3..."]
  ],
  "created_at": 1750000000,
  "id": "...",
  "sig": "..."
}
```

## Client behavior

### Publishing

The event is signed by the social `nsec`, making the announcement
attributable. The `sp` address MAY be derived from any keypair — this
NIP does not mandate a derivation method. Three natural modes:

| Mode | Derivation | Properties |
|------|-----------|------------|
| **Social** | `sp` from social npub directly | Single keypair; address computable from npub by anyone |
| **Deterministic** | `payment_priv = tagged_hash("nostr-payment/vN", nsec)` (N ≥ 1) | Multiple identities from one seed; increment N to rotate while keeping prior identities recoverable |
| **Independent** | Separate keypair | Maximum separation; requires managing two keys |

### Discovering

To find a payment address for a given npub:

1. Query relays: `{"kinds": [30352], "authors": ["<pubkey>"], "#d": ["mainnet"]}`
2. Take the most recent event by `created_at` (relays SHOULD return only the latest, but clients MUST handle duplicates)
3. Read the `sp` tag value
4. Pay using any BIP-352-compatible wallet

If no kind:30352 event exists for the npub, clients MAY fall back to
deriving an `sp1` address from the npub directly (Social mode), with
clear UI disclosure that the receiver has not explicitly announced this
address.

### Rotation

Publishing a new kind:30352 event with the same `d` tag replaces the
previous one at relays. Clients that cache `sp` addresses SHOULD
re-fetch before initiating a payment.

### Verification

If `payment_pubkey` is present, clients MAY verify that the `sp`
address is correctly derived from it using BIP-352 key derivation. If
`payment_pubkey` equals the event author's pubkey, this is a Social
mode derivation and can be verified from the npub alone.

## Rationale

**Why not kind:0?** Kind:0 is a monolithic profile blob. A dedicated
kind allows rotation without republishing the full profile, supports
per-network addresses via the `d` tag, and is cleaner to query, cache,
and index.

**Why not kind:10352 (replaceable)?** Events in the `10000–19999` range
are replaceable per `(pubkey, kind)` — the `d` tag value is not part
of the key. A signet publication would overwrite the mainnet one.
Kind:30352 (addressable) uses `(kind, pubkey, d)` as the key, giving
each network its own independent slot.

**Why not mandate key derivation?** The sender needs only the `sp`
string. How the receiver holds the corresponding scan key — social nsec,
dedicated hardware, multisig — is out of scope for this NIP.

**Receiver culpability.** This NIP does not eliminate the risk that a
receiver exposes scan key material after collecting funds. In
deterministic mode, the scan key is root-equivalent to the social
`nsec`; exposure costs the receiver their Nostr identity. In
independent mode, the receiver can rotate by publishing a new event,
limiting forward exposure.
